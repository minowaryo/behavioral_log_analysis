# review-checklist.md — PRレビューチェックリスト

> `/review` コマンドと合わせて使用する。
> 詳細は `.claude/rules/50-review.md` を参照。

## 作成者セルフチェック（PR作成前）

### 要件・設計
- [ ] `docs/product/use-cases.md` の対応ユースケースに紐づいているか
- [ ] 設計変更がある場合、ADRを作成したか

### コード品質
- [ ] `./vendor/bin/pint` を実行してフォーマットを整えたか
- [ ] `./vendor/bin/phpstan analyse` でエラーがないか
- [ ] Fat Controllerになっていないか
- [ ] N+1クエリがないか（Telescope / Debugbar で確認）

### 認証・認可
- [ ] `$this->authorize()` を適切なControllerで呼んでいるか
- [ ] 新しいエンドポイントに認証ミドルウェアを設定したか

### セキュリティ
- [ ] secrets・APIキーがコードに含まれていないか
- [ ] ログにPIIが出力されていないか
- [ ] バリデーションが適切か（FormRequestを使っているか）

### テスト
- [ ] `php artisan test` が通るか
- [ ] 正常系のFeature Testがあるか
- [ ] 認可（未認証/権限なし）のテストがあるか
- [ ] バリデーションエラーのテストがあるか

### ドキュメント
- [ ] DB変更 → `docs/architecture/data-model.md` を更新したか
- [ ] 設計変更 → 関連docs を更新したか

---

## レビュアーチェック

### 機能・設計
- [ ] 実装がuse-casesの意図に沿っているか
- [ ] 既存の設計パターンと整合しているか
- [ ] 不要なファイルを変更していないか

### セキュリティ
- [ ] 認可チェックが適切か（Policy/Gate経由か）
- [ ] バリデーションが十分か
- [ ] secrets が含まれていないか

### パフォーマンス
- [ ] N+1クエリの疑いがないか
- [ ] 重いクエリはindexが効いているか

### テスト
- [ ] テストが実際に失敗するケースをカバーしているか
- [ ] テスト名が日本語で意図が伝わるか

### ドキュメント
- [ ] 設計変更がdocsに反映されているか
